# Stock Sentinel：当前能力与扩展路线图

更新日期：2026-05-14

## 1. 文档目标
这份文档做两件事：
1. 盘点当前工程里已经可用的功能。
2. 基于当前能力给出下一步推荐扩展，并标注市面上是否已有同类能力的软件。

说明：这里的“市面已有”表示同类能力存在，不代表实现细节与我们完全一致。

## 2. 当前已实现功能清单

| 功能 | 当前实现状态 | 关键代码位置 | 市面已有同类能力 |
| --- | --- | --- | --- |
| 用户登录/注册与会话管理 | 已实现。Supabase Auth + 会话续期 + 受保护路由跳转 | `src/app/login/page.tsx`、`src/lib/supabase/middleware.ts`、`src/proxy.ts` | 有。多数券商与行情平台具备账号体系 |

## 3. 推荐扩展功能（按优先级）

| 推荐功能 | 为什么值得做 | 推荐优先级 | 市面已有同类能力 |
| --- | --- | --- | --- |
| 规则版本化与回滚（Rule Versioning） | 你们的筛选规则会高频变化，必须支持版本记录、灰度、回滚，避免“改规则牵一发而动全身” | P0 | 有。QuantConnect/JoinQuant 等量化平台普遍支持策略迭代与回测流程 |
| 定时扫描 + 主动通知（App/邮件/Webhook） | 从“手动刷新”升级为“系统主动发现机会” | P0 | 有。TradingView、富途均支持条件提醒；TradingView 支持 webhook |
| 历史回测与参数评估 | 让新规则先在历史数据上验证，再决定是否上线 | P0 | 有。JoinQuant、QuantConnect、Ricequant 等 |
| 规则中心（策略注册表） | 支持多套规则并行运行（如保守/激进/短线），避免把策略写死在单一路由 | P1 | 有。量化平台普遍是多策略管理 |
| 信号解释与可审计日志 | 每个信号都能回看“为什么触发”，提升可解释性与复盘效率 | P1 | 有。部分专业平台具备策略日志与审计能力 |
| 回测-实盘一致性检查 | 保障“研究环境”和“线上执行”参数一致，减少行为漂移 | P1 | 有。QuantConnect、JoinQuant 等有类似工程能力 |
| 交易执行连接层（Broker/API） | 把“提醒系统”升级成“半自动/自动执行系统” | P2 | 有。TradingView webhook + 第三方执行器、Futu API 等 |
| 组合层风险控制 | 现在主要是单票规则，后续应支持组合风险预算（仓位上限、行业暴露、相关性） | P2 | 有。机构级量化平台常见 |
| 规则可视化配置台 | 让非开发同学也能修改阈值、启停规则、发布版本 | P2 | 有。部分量化与低代码策略平台提供 |

## 4. 建议的落地顺序（8-12 周）

1. 第 1-2 周：规则版本化 + 统一信号输出协议（先把架构边界定住）。
2. 第 3-4 周：定时扫描与通知（最先提升实际使用价值）。
3. 第 5-8 周：回测框架与规则评估报告（防止策略反复试错）。
4. 第 9-12 周：执行连接层与组合风控（从“看信号”升级到“管执行”）。
