# UI 与股票筛选规则解耦设计方案

更新日期：2026-05-14

## 1. 你关心的问题先给结论

### 1.1 现在的股票筛选策略是客户端做的吗？
不是纯客户端，当前是“服务端为主、客户端有补充计算”的混合模式：

1. 批量筛选主流程在服务端 API：
- `src/app/api/stock/scan/route.ts`
- 服务端调用 `analyzeStock` 计算指标并筛出信号，再返回前端。

2. 客户端也有单票指标计算（用于详情展示和入池辅助）：
- `src/app/stock/[code]/page.tsx`
- 页面拿到 K 线后在浏览器侧调用 `calcTDSequential`、`calcBollinger`。

结论：当前“看板级选股”在后端，“单票展示分析”部分在前端。

### 1.2 你的目标是否合理？
非常合理。  
“规则高频变化，UI相对稳定”这类场景，最佳实践就是把规则引擎做成独立层，UI 只消费统一结果，不感知规则细节。

## 2. 现状痛点（按你目标对照）

1. 规则逻辑散落在多个位置（`scan` 路由 + 个股详情页）。
2. UI 页面知道了太多指标细节，后续改规则容易牵动前端。
3. 规则没有版本化，无法方便做回滚和 A/B 对比。
4. 缺少“规则运行日志”，复盘时很难回答“为什么触发/没触发”。

## 3. 推荐目标架构（建议采用）

## 3.1 分层原则
1. `UI 层`：只负责展示，不写策略判断。
2. `Application 层`：组装流程（取行情 -> 调规则 -> 返回 DTO）。
3. `Strategy Engine 层`：策略接口 + 策略实现 + 版本管理。
4. `Data 层`：行情源、数据库、缓存、调度。

## 3.2 建议目录结构

```txt
src/
  app/
    api/stock/scan/route.ts            # 仅编排，不写规则细节
  modules/
    signal-engine/
      index.ts                         # 入口（runScan）
      types.ts                         # RuleInput/RuleOutput/RuleMeta
      registry.ts                      # 策略注册表（id -> strategy）
      strategies/
        td9_boll_v1/
          strategy.ts
          config.ts
          tests.ts
        td9_boll_v2/
          strategy.ts
          config.ts
          tests.ts
  shared/
    stock-api/
    indicators/
```

## 3.3 统一策略接口（关键）

```ts
export interface ScanStrategy {
  id: string;              // td9_boll_v1
  version: string;         // 1.0.0
  run(input: RuleInput): RuleOutput | null;
}
```

UI 只接受 `RuleOutput`，不关心内部是 TD、BOLL 还是后续新增指标。

## 4. 三种可落地方案（从快到强）

| 方案 | 说明 | 优点 | 风险/代价 | 推荐度 |
| --- | --- | --- | --- | --- |
| A. 代码内策略注册表（同仓库） | 策略模块化 + 版本化，仍随主应用部署 | 改动最小、见效快、最适合当前阶段 | 规则更新仍需发版 | 高（建议先做） |
| B. 数据库配置化规则（DSL） | 阈值和组合条件放 DB，运行时解释执行 | 高频改参数不需要改代码 | 规则解释器复杂度上升 | 中 |
| C. 独立策略服务（微服务） | 规则引擎单独部署，UI/API 通过 HTTP 调用 | 完全解耦，可独立扩容和发布 | 运维与链路复杂度明显增加 | 中（规模上来再做） |

## 5. 建议实施路线（我们可以按这个一起做）

### 阶段 1（1-2 周）：先完成“逻辑解耦”
1. 新建 `signal-engine` 模块与策略接口。
2. 把 `scan/route.ts` 内规则迁移到 `strategies/td9_boll_v1`。
3. `scan` API 只保留编排逻辑，返回统一 `RuleOutput`。
4. 前端页面禁止直接依赖规则细节字段（只读输出 DTO）。

### 阶段 2（1-2 周）：做“版本管理和可审计”
1. 新增策略版本号（`strategy_id`, `strategy_version`）。
2. 每次扫描写入运行日志表（触发数量、耗时、命中样本）。
3. 支持按版本切换运行（环境变量或后台配置）。

### 阶段 3（2-4 周）：做“稳定运行能力”
1. 增加定时任务调度（Vercel Cron 或 Supabase pg_cron + Edge Functions）。
2. 增加通知通道（站内、邮件、Webhook）。
3. 增加回测入口，形成“研究 -> 上线 -> 复盘”的闭环。

## 6. 和当前代码的映射关系（迁移清单）

1. 保留并复用：
- `src/lib/indicators/*`（指标计算函数）
- `src/lib/stock-api/*`（行情接入）

2. 需要迁移：
- `src/app/api/stock/scan/route.ts` 的筛选判断代码迁到 `signal-engine/strategies/*`
- `src/app/stock/[code]/page.tsx` 中策略判断部分改为“调用统一分析结果”或“仅图表渲染”

3. 需要新增：
- `signal-engine` 模块
- `strategy_runs`（运行日志表，可选）
- `strategy_versions`（版本配置表，可选）

## 7. 推荐先做的最小闭环（MVP of Decoupling）

1. 先落地方案 A（同仓库策略注册表）。
2. 只抽离一个策略 `td9_boll_v1`，先跑通。
3. 保持 UI 不改样式，只改数据来源。
4. 用 2-3 个属性测试覆盖策略输入输出稳定性。

做到这一步，你就能获得：
- 改规则不动 UI。
- 规则可以迭代版本。
- 风险可控，改造成本低。
